# ブランチコードレビュワー

派生もとからの変更差分、コミット履歴、変更差分に関連する情報をもとにコードレビューを行う。

## ワークフロー

進捗に合わせてチェックを入れてください：

```
PR操作の進捗:
- [ ] ステップ1: 派生元のコミットハッシュを取得
- [ ] ステップ2: 最終的な変更差分を取得
- [ ] ステップ3: コミット履歴を取得
- [ ] ステップ4: コードレビューを開始する
- [ ] ステップ5: コードレビュー結果を生成する
- [ ] ステップ6: コードレビュー結果のセルフレビュー
- [ ] ステップ7: 最終的なコードレビュー結果を出力する
```

### ステップ1: 派生元のコミットハッシュを取得

以下のコマンドを実行する。

```sh
git merge-base BASE_BRANCH HEAD
```

ユーザーから分岐元のブランチ名が明示されていない場合、
「派生元のブランチ名が必要です。」と返答し終了する。


### ステップ2: 最終的な変更差分を取得

以下のコマンドを実行する

```sh
git diff 派生元のコミットハッシュ
```

### ステップ3: コミット履歴を取得

以下のコマンドを実行する

```sh
git log -p 派生元のコミットハッシュ..HEAD
```

### ステップ4: コードレビューを開始する

ステップ2、ステップ3で取得した情報をもとにレビューを開始する。

- 変更差分だけでは理解できない場合は関連コードも適時読み込むこと
  （eg: 変更差分で定数を使用していて、定数は変更差分に含まれておらず、実際の値が重要な場合、定数が定義されているファイルを読み込む）
- 外部ライブラリを使用している場合、必要に応じて外部ライブラリの情報を取得し、正しい知識をもとにレビューすること

### ステップ5: コードレビュー結果を生成する

- レビュー項目には重要度をつけること（低、中、高）
- 指摘事項には常に代案とトレードオフについての考察を含めること

### ステップ6: コードレビュー結果のセルフレビュー

ステップ5 で生成したコードレビュー結果をセルフレビューし、
不適当な内容が含まれていないか、指摘漏れがないかをチェックし、
必要であれば修正を行う。


### ステップ7: 最終的なコードレビュー結果を出力する

- 重要度でグルーピングすること
- 重要度順に並び替えること
