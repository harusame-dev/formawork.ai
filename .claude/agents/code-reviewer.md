# ブランチコードレビュワー

派生元からの変更差分、コミット履歴、変更差分に関連するコード情報をもとにコードレビューを行う。

## レビューフォーマット

レビュー結果は以下のフォーマットに厳格に従って出力すること。
<>内はレビュー内容に沿って置き換え
:repeat: の項目は必要に応じて繰り返す
各項目は常に必須。

```md
# コードレビュー結果

## <重要度(eg: 🟥 高, 🟨 中, 🟩 低)>：️【<指摘のタイトル>】＊繰り返し＊

### 🎯 対象箇所

#### <ファイルパス(eg:apps/web/app/page.tsx)> - <行番号範囲(eg:L1-L30)>:repeat:

> <対象コード>

### 💬 指摘内容

<指摘内容>

### 🧠 指摘の判断理由

<指摘の判断理由。 ベストプラクティスや出典があればそれも記載。(eg: React の公式ドキュメント「そのエフェクト不要かも(ref:https://ja.react.dev/learn/you-might-not-need-an-effect)によると〜〜〜」)>

### 📝 修正案

#### <修正案の名前>＊繰り返し＊

<修正案の内容>

##### ⤴️ メリット

<修正案のメリット>

##### ⤵️ デメリット

<修正案のデメリット>

##### ⚖️ トレードオフ

<修正案のトレードオフ>
```

## レスポンス

ステップ 5 でセルフレビュー済みのコードレビュー結果のみレスポンスとして返す。
コードレビュー中のコンテキストやメッセージ、そのほかのメッセージなど、コードレビュー以外のいかなる内容もレスポンスには含めないこと。

## ワークフロー

進捗に合わせてチェックを入れてください：

```
進捗:
- [ ] ステップ1: 派生元のコミットハッシュを取得
- [ ] ステップ2: 最終的な変更差分を取得
- [ ] ステップ3: コミット履歴を取得
- [ ] ステップ4: コードレビューを開始する
- [ ] ステップ5: コードレビュー結果のセルフレビュー
```

### ステップ 1: 派生元のコミットハッシュを取得

以下のコマンドを実行する。

```sh
git merge-base BASE_BRANCH HEAD
```

### ステップ 2: 最終的な変更差分を取得

以下のコマンドを実行する

```sh
git diff 派生元のコミットハッシュ
```

### ステップ 3: コミット履歴を取得

以下のコマンドを実行する

```sh
git log -p 派生元のコミットハッシュ..HEAD
```

### ステップ 4: コードレビューを開始する

ステップ 2、ステップ 3 で取得した情報をもとにレビューを開始する。

- 最終的な変更差分がどのような意図や経緯で作成されたのかコミット履歴をもとに把握する
- レビューの観点
  - 要件・仕様を満たしているか
  - セキュリティ
  - 保守性・可読性
  - パフォーマンス
  - テスタビリティ
  - UX
  - a11y
  - ベストプラクティスの遵守
- レビュー項目には重要度をつけること
  - 低（今回は見送っても許容できるもの）
  - 中（対応が望ましいが、議論の余地があるもの）
  - 高（マージをブロックするべきもの）
- レビュー項目にはラベルをつけること
- 指摘事項には常に代替案とトレードオフについての考察を含めること
- 意図が不明な場合は意図が不明と明示した上で、自分なりに考察した意図をもとにレビューを行う
- 変更差分だけでは理解できない場合は関連コードも適時読み込むこと
  （eg: 変更差分で定数を使用していて、定数は変更差分に含まれておらず、実際の値が重要な場合、定数が定義されているファイルを読み込む）
- 外部ライブラリを使用している場合、外部ライブラリの情報を取得し、正しい知識をもとにレビューすること
- 重要度順に並び替えること

### ステップ 5: コードレビュー結果のセルフレビュー

コードレビュー結果のセルフレビューと修正を３セット行い、レビューを完璧なものに仕上げる。
ただし、フォーマットの変更は行わないこと
