# コードレビュースキル

コードレビュー時のフォーマット、観点、重要度基準を提供する。

## レビューフォーマット

レビュー結果は以下のフォーマットに厳格に従って出力すること。
`<>`内はレビュー内容に沿って置き換える。
`*繰り返し*` の項目は必要に応じて繰り返す。
各項目は常に必須。

```md
# コードレビュー結果

## <重要度(eg: 🟥 高, 🟨 中, 🟩 低)>：️【<指摘のタイトル>】*繰り返し*

### 🎯 対象箇所

#### <ファイルパス(eg:apps/web/app/page.tsx)> - <行番号範囲(eg:L1-L30)>*繰り返し*

> <対象コード>

### 💬 指摘内容

<指摘内容>

### 🧠 指摘の判断理由

<指摘の判断理由。 ベストプラクティスや出典があればそれも記載。(eg: React の公式ドキュメント「そのエフェクト不要かも(ref:https://ja.react.dev/learn/you-might-not-need-an-effect)によると〜〜〜」)>

### 📝 修正案

#### <修正案の名前>*繰り返し*

<修正案の内容>

##### ⤴️ メリット

<修正案のメリット>

##### ⤵️ デメリット

<修正案のデメリット>

##### ⚖️ トレードオフ

<修正案のトレードオフ>
```

## レビュー観点

- 要件・仕様を満たしているか
- セキュリティ
- 保守性・可読性
- パフォーマンス
- テスタビリティ
- UX
- a11y
- ベストプラクティスの遵守

## 重要度基準

レビュー項目には重要度をつけること：

- 🟩 低: 今回は見送っても許容できるもの
- 🟨 中: 対応が望ましいが、議論の余地があるもの
- 🟥 高: マージをブロックするべきもの

## レビュー時の注意事項

- 指摘事項には常に代替案とトレードオフについての考察を含めること
- 意図が不明な場合は意図が不明と明示した上で、自分なりに考察した意図をもとにレビューを行う
- 変更差分だけでは理解できない場合は関連コードも適時読み込むこと
  （eg: 変更差分で定数を使用していて、定数は変更差分に含まれておらず、実際の値が重要な場合、定数が定義されているファイルを読み込む）
- 外部ライブラリを使用している場合、外部ライブラリの情報を取得し、正しい知識をもとにレビューすること
- 重要度順に並び替えること
