## ワークフロー

進捗に合わせてチェックを入れてください：

```
バリデーション進捗:
- [ ] ステップ1: バリデーションチェック実行
- [ ] ステップ2: エラー確認
- [ ] ステップ3: 自動修正実行（エラー時）
- [ ] ステップ4: 再チェック実行（エラー時）
- [ ] ステップ5: エラー分析（残存エラー時）
- [ ] ステップ6: 修正方法の提示
```

## ステップ1: バリデーションチェック実行

```bash
pnpm -w validate:check
```

**チェック対象:**
- **lint**: Biome による構文・規約チェック
- **format**: Biome による整形チェック
- **デッドコード**: Knip による未使用コード検出
- **スペル**: Knip によるスペルチェック
- **型**: TypeScript / Next.js build による型チェック

## ステップ2: エラー確認

出力を分析して以下を確認：

- エラーの有無
- エラー数（件数別）
- エラー種別（lint、format、deadcode、spell、type）
- 該当ファイル・行番号

## ステップ3: 自動修正実行（エラー時）

エラーが存在する場合：

```bash
pnpm -w validate:fix
```

**自動修正対象:**
- lint エラー（Biome 対応分）
- format エラー（Biome）
- デッドコード（Knip 対応分）

**自動修正対象外:**
- スペルエラー（個別対応が必要）
- 型エラー（コード修正が必要）

## ステップ4: 再チェック実行（エラー時）

自動修正実行後、再度バリデーションを実行：

```bash
pnpm -w validate:check
```

## ステップ5: エラー分析（残存エラー時）

自動修正後も残存するエラーについて詳細分析：

**分析対象:**
- スペルエラーの具体的な箇所
- 型エラーの原因と修正方法
- 複数エラーの関連性

**分析時のリソース:**
- プロジェクトの CLAUDE.md コーディング規約
- Context7 によるライブラリドキュメント参照
- 関連するコード箇所の確認

## ステップ6: 修正方法の提示

エラー分析結果を以下フォーマットで報告：

```
## バリデーション結果

### エラー概要
- lint エラー: X 件
- format エラー: X 件
- デッドコード: X 件
- スペルエラー: X 件
- 型エラー: X 件

### エラー詳細

#### 1. ファイルパス:行番号 [エラー種別]
- **エラーメッセージ**: [生メッセージ]
- **エラー内容**: [わかりやすい説明]
- **エラーの原因**: [なぜ発生しているか]
- **対応方法**: [具体的な修正内容]
- **対応理由**: [なぜこの修正が必要か]

#### 2. ...
```

**報告時のソート順:**
1. ファイルパス
2. エラー種別
3. 行番号

**報告時のルール:**
- エラー無視は避け、修正を最優先
- ignore する場合はコメントで理由を記載
- 複数エラーの関連性を説明
- any 型使用は最大限避ける

## パス（バリデーション成功時）

```
✅ すべてのバリデーションがパスしました

チェック内容:
- lint: OK
- format: OK
- デッドコード: OK
- スペル: OK
- 型: OK
```

## 成功時のメッセージ

- **すべてパス**: 「すべてのバリデーションがパスしました」と報告
- **自動修正で解決**: 修正内容と再チェック結果を報告
- **手動修正が必要**: 詳細分析結果と修正方法を報告

## エラーハンドリング

| 状況 | 対応 |
|-----|-----|
| バリデーション実行エラー | エラーメッセージを表示、原因を特定 |
| 自動修正実行エラー | 修正可能なエラーリストと理由を提示 |
| 型チェック失敗 | TypeScript エラー内容を詳細分析 |

## 品質基準

コードベースが以下を満たすことを確認：

- lint エラー: 0 件
- format エラー: 0 件
- デッドコード: 0 件
- スペルエラー: 0 件
- 型エラー: 0 件
