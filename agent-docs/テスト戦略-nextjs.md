# テスト戦略（Next.js）

[テスト戦略.md](./テスト戦略.md) の方針を Next.js プロジェクトに適用するうえでの、Next.js 固有の取り扱いをまとめる。

## サーバーコンポーネントの扱い

### 方針

- Next.js のサーバーコンポーネント（React Server Components）は **E2E テストで動作を担保** する
- サーバーコンポーネントのファイルは **E2E 以外のカバレッジ計測の対象に含めない**
- サーバーコンポーネント内のロジック（データ取得・データ整形・条件分岐・計算など）は **極力外部ファイルの関数に切り出し、切り出した関数をテスト対象とする**

### 理由

- **エコシステムの制約**: Next.js のサーバーコンポーネントは async コンポーネントやサーバー専用 API、RSC ペイロードの解決を前提としており、Vitest や Jest など既存のテストランナーから単体で実行することが現状困難。公式・準公式に確立されたテスト手段が揃っていないため、サーバーコンポーネントそのものを E2E 以外でテストしようとすると、過度なモックや独自セットアップで脆く保守困難なテストになりやすい
- **カバレッジに含めると基準が歪む**: テスト不能なファイルをカバレッジ対象に含めると、ベースラインが下振れし、80% という閾値が機能しなくなる。エコシステム由来でテスト不能な領域は明示的に除外し、テスト可能な範囲の網羅率として基準を運用する
- **ロジックを切り出せば通常のテストで担保できる**: サーバーコンポーネントを「データ取得とレンダリングの薄い層」に保ち、計算・分岐・整形のロジックを純粋関数として外部に切り出せば、その関数は Small / Medium のテストで網羅できる。サーバーコンポーネント自体は E2E で表示確認すれば十分になる

### 実装方法

#### ロジックの切り出し

サーバーコンポーネントのファイルには「データ取得（`fetch` や DB アクセス）」「切り出した関数の呼び出し」「JSX の出しわけ・レンダリング」のみを残し、ロジックは別ファイルに分離する。

#### E2E での担保

- サーバーコンポーネントが描画されるページに対しては、主要なユーザーシナリオの中で表示・遷移を E2E で確認する
- サーバーコンポーネント固有の表示分岐（ログイン状態による出し分け、権限による出し分けなど）は E2E のシナリオに含める
