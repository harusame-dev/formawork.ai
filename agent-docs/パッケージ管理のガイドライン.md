# パッケージ管理のガイドライン

## 概要

パッケージ管理におけるセキュリティ・品質・メンテナンス性を統一するためのガイドライン。

## 1. インストール可能なパッケージの公開日数

### 方針

パッケージ公開から 14 日以上経過していないパッケージのインストールを禁止する。

### 理由

- 悪意のあるパッケージは通常数時間以内に検知・削除されるため、インストールを遅らせるだけで、コンパイル時に汚染版を踏むリスクを大幅に下げられる
- 14 日という数字は Renovate で automerge をする場合に推奨されている値で、レジストリが悪意のある依存関係を削除する時間を十分に確保できる
- メジャーアップデート以外は基本的に automerge 前提のため、automerge 推奨値である 14 日に揃える

### 実装方法

#### pnpm

`pnpm-workspace.yaml` で `minimumReleaseAge` を設定する（pnpm 11 以降、`pnpm-workspace.yaml` が正式な設定箇所）。

```yaml
# pnpm-workspace.yaml
minimumReleaseAge: 20160
```

`minimumReleaseAge` の単位は分のため、14 日 = `20160` 分を指定する。

#### Renovate

`renovate.json` に `minimumReleaseAge` を設定する。

```json
{
  "extends": ["config:recommended"],
  "minimumReleaseAge": "14 days"
}
```

#### Go

Go には `minimumReleaseAge` に相当する機能が存在しないため、CI で `gomod-age` によるチェックワークフローを必ず作成する。

ただし、ローカルでの `go get` などによるインストールを CI では防げない。根本的な対策が打てないため、新しいパッケージを導入する際はリリースから 14 日以上経過しているバージョンを指定すること。

## 2. JavaScript パッケージのバージョン指定

### 方針

`package.json` ではバージョン範囲指定（`^1.2.3` や `~1.2.3`）ではなく、exact version（`1.2.3`）でインストールすること。

### 理由

Renovate が推奨しており、以下のメリットがあるため。

- **何が動いているかが明確になる**: `package.json` に固定バージョンが書かれていれば、誰がいつインストールしても同じバージョンが入る。範囲指定の場合は、インストール時期やロックファイルの状態によって実際に入るバージョンが変わるため、何が動いているかが分かりづらくなる
- **障害発生時の切り分けが容易になる**: 不具合のあるバージョンがリリースされた場合、範囲指定だと知らないうちに本番に取り込まれて原因特定が難しくなる。exact version であれば、バージョンを上げた PR が CI で失敗し、その PR をマージしない判断ができるため、`main` ブランチや本番環境を壊さずに済む
- **更新が PR として明示される**: バージョンアップは必ず Renovate の PR として現れるため、変更内容・タイミング・レビュー履歴が残り、変更管理が一貫する

### 実装方法

#### pnpm

`pnpm-workspace.yaml` で `savePrefix` を空文字に設定する（pnpm 11 以降、`pnpm-workspace.yaml` が正式な設定箇所）。

```yaml
# pnpm-workspace.yaml
savePrefix: ""
```

`savePrefix` のデフォルトは `^`。空文字を指定することで `pnpm add` 実行時に `^` や `~` が付かない exact version で `package.json` に記録される。

#### Renovate

`renovate.json` で `config:pinAllExceptPeerDependencies` を有効化する。

```json
{
  "extends": ["config:recommended", "config:pinAllExceptPeerDependencies"]
}
```

`peerDependencies` 以外のすべての依存関係を exact version にピン留めする。既存の範囲指定もピン留めする PR が自動で作成される。

## 3. パッケージの自動更新

### 方針

パッケージの自動更新を必ず導入すること。

### 理由

- **セキュリティ脆弱性への追従コストを下げる**: 脆弱性が公表されてから手動で更新版を確認・適用するのは追従漏れが発生しやすい。自動化により脆弱性修正版が PR 化されるため、対応のリードタイムを短くできる
- **手動更新の運用負荷を下げる**: 依存関係が増えるほど手動更新は現実的でなくなる。検知・PR 作成までが自動化されるため、レビューとマージに集中できる
- **依存関係の鮮度を保てる**: 更新が後回しになると一度に大量のメジャーアップデートを処理することになり、破壊的変更の影響範囲が広がる。継続的に小さく更新することで、変更の影響を局所化できる

### 実装方法

[Renovate](https://docs.renovatebot.com/) を導入する。

## 4. JavaScript パッケージマネージャー

### 方針

JavaScript のパッケージマネージャーは pnpm を利用する。

### 理由

- ハードリンクと content-addressable store によりディスク使用量とインストール時間を削減できる
- workspace 機能や catalog 機能などモノレポ運用に有用な機能が標準で提供されている
- npm や yarn の classic と異なり、デフォルトで厳密な依存解決（phantom dependency を許容しない）を行い、宣言されていない依存への暗黙の参照を防げる

## 5. パッケージマネージャーのバージョン統一

### 方針

パッケージマネージャーのバージョンはチームで揃えること。

### 理由

- バージョン差異による挙動の違いや、lockfile の再生成・差分発生を防げる
- ローカル・CI 間で同じバージョンを使うことで、再現性のあるインストール結果を担保できる

### 実装方法

プロジェクトルートの `package.json` の `packageManager` フィールドに `pnpm@<バージョン>` を記載する。

```json
{
  "packageManager": "pnpm@10.0.0"
}
```

Corepack によって `packageManager` フィールドに記載されたバージョンが自動的に利用される。

## 6. モノレポにおける依存バージョンの統一

### 方針

モノレポを構成する場合、各パッケージで同じ外部ライブラリを使用するときはバージョンを揃えること。

### 理由

みt
- パッケージごとに異なるバージョンが入ると、同一ライブラリの複数バージョンが混在し、バンドルサイズ増加・型不整合・実行時の挙動差異の原因になる
- バージョンを揃えることで、更新時の影響範囲を一度に把握でき、検証コストを削減できる
- 依存関係の重複インストールを避けることで、インストール時間とディスク使用量を削減できる

### 実装方法

pnpm の [catalog](https://pnpm.io/catalogs) 機能を使用する。

`pnpm-workspace.yaml` に `catalog` を定義し、各パッケージの `package.json` では `catalog:` プロトコルで参照する。

```yaml
# pnpm-workspace.yaml
catalog:
  react: 19.0.0
  react-dom: 19.0.0
```

```json
// 各パッケージの package.json
{
  "dependencies": {
    "react": "catalog:",
    "react-dom": "catalog:"
  }
}
```

`catalog:` を指定したパッケージは、`pnpm-workspace.yaml` に定義された単一のバージョンに解決される。

また、`pnpm-workspace.yaml` で `catalogMode` を `strict` に設定し、カタログ経由でのバージョン管理を強制する。

```yaml
# pnpm-workspace.yaml
catalogMode: strict
```

`strict` を指定すると、`pnpm add` 実行時にカタログへ自動登録され、カタログ外のバージョンを追加しようとするとエラーになる。これによりカタログを経由しないバージョン追加を防ぎ、モノレポ全体での依存バージョン統一を機械的に担保できる。

## 7. インストールスクリプトの禁止

### 方針

依存パッケージの `preinstall` / `postinstall` などのライフサイクルスクリプトの実行は原則禁止する。

### 理由

- インストール時に任意コードが実行されるため、サプライチェーン攻撃の主要な経路になる
- 悪意あるパッケージは `postinstall` で認証情報の窃取・バックドア設置などを行うことが多く、実行を抑止するだけで攻撃面を大幅に削減できる
- ネイティブビルドが必要な一部のパッケージを除き、スクリプト実行を許可しなくても通常の開発・実行に支障はない

### 実装方法

#### pnpm

pnpm 10 以降は依存パッケージのライフサイクルスクリプトをデフォルトで実行しないため、追加の設定は不要。

加えて、`pnpm-workspace.yaml` で `allowBuilds` を `false` に設定し、ビルドスクリプトの実行を明示的に禁止する。

```yaml
# pnpm-workspace.yaml
allowBuilds: false
```

どうしても実行が必要なパッケージがある場合は、`onlyBuiltDependencies` に個別に列挙してレビューを経たうえで許可する。
