# テスト戦略

## 概要

自動テストの粒度・構造・実装スタイルを統一し、信頼性と保守性の高いテストスイートを構築するための戦略。

## 1. テストサイズの分類

### 方針

Google のテストサイズ ([Small / Medium / Large](https://testing.googleblog.com/2010/12/test-sizes.html)) の考え方を採用し、各テストがどのサイズに該当するかを意識して設計・実装する。

| サイズ   | 実行範囲                                       | 外部依存                                                                  | 実行時間の目安 |
| -------- | ---------------------------------------------- | ------------------------------------------------------------------------- | -------------- |
| Small    | 単一プロセス・単一スレッド内で完結             | 不可（ネットワーク・ファイル I/O・DB・スリープ・マルチスレッドはすべて不可） | 数十ミリ秒以内 |
| Medium   | 単一マシン内で完結                             | localhost に閉じたネットワーク、ファイル I/O、ローカル DB は可            | 数秒以内       |
| Large    | 複数マシン・外部システムを含む                 | 外部 API、外部 DB、ブラウザ越しの E2E など制約なし                        | 数秒〜数分     |

### 理由

- テストサイズを共通言語にすることで、各テストの依存範囲・実行コスト・期待される安定性をチーム全員が同じ尺度で議論できる
- サイズごとに実行頻度・並列度・タイムアウト設定を分けられ、CI のフィードバックループを最適化できる
- 「単体テスト」「結合テスト」のような曖昧な分類より、依存範囲という客観的な基準で分けられるため、判断のブレが少ない

## 2. テストサイズの優先順位

### 方針

Medium サイズのテストをベースとして優先的に実装する。Small は補助的に、Large は最小限に留める。

### 理由

- **Medium を主軸にする理由**: 実プロダクトに近い構成（ローカル DB やローカルサーバーを含む）で検証できるため、結合バグやスキーマ不整合などプロダクトの破壊につながりやすい不具合を高い確度で検知できる。Small だけでは過剰なモックに依存し、リファクタ耐性が低く偽陽性・偽陰性が増える
- **Small を補助にする理由**: 純粋ロジック（バリデーション、計算、変換）は Small で網羅した方が高速かつ安定する。ただし Small は「依存を全てモックした結果、本番経路を通っていない」テストになりやすいため、振る舞いの保証ではなくロジックの網羅に用途を限定する
- **Large を最小限にする理由**: 実行時間・安定性・メンテナンスコストすべてで不利なため、ユーザー操作シナリオなど Medium で代替不能な範囲に絞る。Large でしか守れない契約以外は Medium に押し下げる


## 3. テストの構造

### 方針

テストはグルーピングを行わず、フラットな構造で記述する。


### 理由

- **テスト名だけで意図が伝わる**: フラット構造ではテスト名に状況と期待結果を完結に書く必要があるため、テスト単体で意味が読み取れる。ネスト構造ではタイトルがグループ名とテスト名に分断され、失敗ログだけでは文脈の再構築にコストがかかる
- **共有状態の混入を防げる**: グルーピングはライフサイクルフックと組み合わさってネスト固有の暗黙の前提を生みやすい。フラット構造はそのような共有状態の入り込む余地がなく、テストごとの独立性が高まる
- **テスト一覧の並び替え・抽出が容易**: 階層に縛られず、テスト名の命名規約だけでフィルタや並び替えができる


## 4. 事前準備・後処理の実装方法

### 方針

ファイル全体やテスト群に暗黙的に適用されるライフサイクルフック（テスト前後・スイート前後の一括フック）は使用せず、テストが必要とする前提を引数として明示的に受け取るフィクスチャ形式で実装する。事前準備（データ作成・依存リソースの起動）と後処理（データ削除・リソースの解放）はフィクスチャ側に閉じ込め、テスト本体は検証対象の振る舞いの記述に集中させる。

### 理由

- **テストごとに必要な前提だけを宣言できる**: フィクスチャは引数として受け取った場合のみセットアップが走るため、テストごとに必要なものを宣言的に指定できる。一括フックのように「ファイル内のすべてのテストに同じ前提が暗黙に適用される」状態を避けられる
- **後処理の漏れがなくなる**: フィクスチャはセットアップとクリーンアップを 1 つの構造として記述するため、テストが失敗・スキップされてもクリーンアップが必ず実行される。後処理フックの書き忘れ・対象漏れによるリソースリークを防げる
- **テスト本体の見通しが良くなる**: テストファイル冒頭にライフサイクルフックが並ばないため、テスト本体には「何を検証しているか」だけが残り、第三者が読んだときの認知負荷が下がる
- **再利用と組み合わせが容易**: フィクスチャは関数として export でき、複数のテストファイル間で再利用できる。特定のファイル・スコープに紐付かない

### 実装方法

- フィクスチャは「テストが必要とする前提」を表現するものとし、テスト内で実施する操作（クリック、API 呼び出しの検証など）は含めない
- 1 つのフィクスチャに複数の責務を詰め込まず、フィクスチャ間の依存で組み立てる
- 共有可能なフィクスチャは専用ディレクトリ等に切り出し、複数のテストファイルから再利用する
- Vitest および Playwright を使用する場合は、それぞれが提供する [`test.extend`](https://vitest.dev/guide/test-context.html#test-extend) ([Playwright 版](https://playwright.dev/docs/test-fixtures)) を用いてフィクスチャを定義する。`beforeAll` / `beforeEach` / `afterAll` / `afterEach` は使用せず、セットアップ・クリーンアップはフィクスチャ関数内（`use` の前後）に記述する
- それ以外のテストランナーを使う場合は、ランナーがフィクスチャ機構を提供していればそれを利用し、無ければ等価な構造（セットアップとクリーンアップを 1 つの単位として記述し、テストが必要なものだけを取り出せる形）を採用する

## 5. モックの扱い

### 方針

- **モックは極力避ける**。本物の依存（ローカル DB、ローカルサーバー、実際の関数）を使えるなら本物を使う
- モックを使うのは、本物を使うことが現実的に不可能・不適切な場合（外部 API、課金が発生するサービス、非決定性のある時計・乱数、長時間かかる処理など）に限定する

### 理由

- **モックは「テストは通るのに本番では壊れる」状況を生む**: モックはテスト作者が想定した振る舞いを再現するだけで、本物の依存の実際の挙動とは乖離する。スキーマ変更・契約変更・エラー応答などモック側に反映漏れがあると、テストはグリーンでも本番では壊れる
- **モックは保守コストを生む**: 依存先の仕様変更に追従してモックも更新し続ける必要があり、本物を使えば不要なメンテナンス作業が常時発生する
- **Medium 主軸の戦略と整合する**: 本物のローカル DB・ローカルサーバーを使う Medium テストは、そもそもモックを必要としない構成になっており、本方針と自然に噛み合う

### 実装方法

#### 本物を使えるものは本物を使う

- DB アクセスはローカル DB（Testcontainers・ローカルの Postgres / SQLite など）を使う
- HTTP リクエストはローカルサーバー（テスト用に立てた実サーバー）を使う
- ファイル I/O はテンポラリディレクトリを使う
- 同一プロセス内の関数・モジュールは直接呼ぶ（依存注入のためだけの抽象化やモックは作らない）

#### モックを許容するケース

- 外部 API（呼び出しに費用・時間・副作用が発生するもの）
- 時計・乱数・UUID 生成など、テストで決定性が必要な非決定的要素
- 本物を起動するコストが著しく高い、もしくはローカルで再現不可能な依存

これらの場合も、可能な限り **公式に提供されているフェイク実装**（例: Stripe の Test Mode、AWS の LocalStack、ライブラリ提供の in-memory 実装）を優先し、自前のモックは最後の手段とする。

## 6. 振る舞いベースのテスト

### 方針

**内部実装ではなく、振る舞い（入力に対する出力・観測可能な副作用）をテストする**。テスト対象がどのように動くかではなく、何をするかを検証する。

### 理由

- **内部実装に依存したテストはリファクタ耐性を奪う**: 「この関数がこの順番で呼ばれた」「この private メソッドが何回呼ばれた」といったテストは、内部構造の変更だけでテストが落ちる。振る舞いが同じなら通るテストにしておくことで、安全にリファクタできる
- **振る舞いをテストすればテストがドキュメントになる**: 「この入力でこの出力になる」「この操作後に観測可能な状態がこうなる」という記述は、利用者視点での仕様そのもの。実装の都合（呼び出し順・内部状態）を書いたテストは仕様としては読めない
- **本来検知すべき不具合を見逃さない**: 内部実装をテストしていると、内部構造は維持されたまま外部から見た振る舞いが壊れるケースを検知できない。観測可能な結果を検証することで、利用者が遭遇する不具合に直結したカバレッジになる

### 実装方法

- 検証対象は「テスト後に観測できる状態（DB の中身、ファイルの内容、レスポンス、送信されたメッセージ）」にする
- 「何が呼ばれたか」ではなく「何が起きたか」を書く
- どうしても呼び出しを検証したい場合（外部 API への副作用など）は、API のフェイクサーバーが受け取ったリクエストを検証するなど、観測可能な境界で確認する

## 7. コンポーネントテスト・E2E テストでの要素操作

### 方針

コンポーネントテストおよび E2E テストでは、アクセシビリティクエリ（ロール・アクセシブルネーム・ラベルなど、支援技術が認識する属性）を用いて要素を取得・操作する。テスト ID・CSS セレクタ・XPath などの実装詳細に依存したクエリは原則使用しない。

### 理由

- **振る舞いベースのテスト方針と整合する**: アクセシビリティクエリは「ユーザー（および支援技術）が画面をどう認識するか」を基準に要素を特定するため、利用者視点での振る舞いをそのまま検証することになる
- **アクセシビリティの担保につながる**: テストが通る条件として適切なロール・ラベルが必要になるため、アクセシビリティ違反がテスト時点で表面化する。テスト ID 等でバイパスしていると、支援技術で使えない UI が放置されやすい
- **リファクタ耐性が高い**: DOM 構造・class 名・コンポーネント階層の変更でテストが落ちにくい。ロールや名前は実装ではなくユーザー体験に紐づくため、変わりにくい

### 実装方法

- ボタン・リンク・フォーム要素などは「ロール + アクセシブルネーム」で取得する
- 入力欄はラベルで取得する
- 一覧項目などはロールと表示テキストで特定する
- 適切なクエリで取得できない場合は、まずマークアップを見直してアクセシビリティを改善する（適切な要素・ロール・ラベルを与える）
- どうしても識別不能な場合に限り、最終手段として専用属性（`data-testid` 等）を使う
