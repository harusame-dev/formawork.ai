# テスト戦略（カバレッジ）

[テスト戦略.md](./テスト戦略.md) の方針を補完する、カバレッジに関する戦略をまとめる。

カバレッジの方針はテストサイズではなく、テストの目的軸（E2E か、そうでないか）で定義する。サイズはテストの構成（プロセス・マシン・外部依存の範囲）の分類であり、E2E かどうかとは独立した概念であることに注意する。

## 1. 測定対象

### 方針

- **E2E テスト以外のテスト**のカバレッジを測定する

### 理由

- **E2E はシナリオ単位での価値検証が主目的**: 「ユーザーが達成したい目的を end-to-end で実行できるか」を保証することが E2E の主たる役割であり、コード網羅率はその目的に対する適切な指標ではない
- **サーバーサイドのコードのカバレッジ測定が困難**: E2E はブラウザ越しに本番に近い構成のサーバーを叩く形になるため、フロントとサーバーで別プロセス・別ランタイムにまたがるカバレッジを収集・統合する必要がある。インストルメンテーションの仕込み、複数プロセスからのカバレッジ情報の収集、フロントとサーバーのレポート統合といった手順が必要で、計測パイプラインの構築・維持コストが大きい
- **重複した網羅はコストに見合わない**: 同じコードパスを E2E と下層のテストで二重に網羅しても、検出できる不具合の増分は小さい一方、実行コストは大きく増える

## 2. 達成基準

### 方針

E2E 以外のテストにおいて、**ブランチカバレッジ（Branch Coverage / C1）で 80% 以上** を維持する。

### 理由

- **ステートメント / 行カバレッジでは式レベルの分岐を取りこぼす**: 三項演算子・`&&` / `||` の短絡評価・`?.` / `??` などは、文単位の指標では実行さえされれば 100% になってしまう。ブランチカバレッジまで見ることで、これらの片側経路の取りこぼしを検知できる
- **80% は実務的なバランスライン**: 100% を目標にするとテストのためのテストや到達困難なエラーパスのカバーに労力が割かれ、費用対効果が悪化する。一方 50〜60% 台では主要分岐の取りこぼしが常態化する。80% を下限にすることで、重要分岐の網羅を担保しつつ、過剰な追従コストを避けられる
