# テストの使い分け

## テスト種別の概要

| 種別                  | ファイル拡張子                  | ツール                   | 外部依存       |
| --------------------- | ------------------------------- | ------------------------ | -------------- |
| E2E テスト            | `*.small.e2e.test.ts`           | Playwright               | ブラウザ + DB  |
| small ブラウザテスト  | `*.small.browser.test.{tsx,ts}` | Vitest（ブラウザモード） | なし（モック） |
| medium サーバーテスト | `*.medium.server.test.ts`       | Vitest（サーバーモード） | DB あり        |
| small サーバーテスト  | `*.small.server.test.ts`        | Vitest（サーバーモード） | なし（モック） |

## 判断フロー

```
テストしたい内容は何か？
├─ 複数レイヤーにまたがる機能フロー（ページ遷移・DB 書き込み含む）
│    → E2E テスト
├─ クライアントコンポーネント / React Hook の UI 動作
│    → small ブラウザテスト
├─ サーバーロジック（DB への実際のアクセスが必要）
│    → medium サーバーテスト
└─ サーバーロジック（DB アクセスなし：権限・バリデーション・エラーハンドリング）
     → small サーバーテスト
```

## E2E テスト

**対象：** ページ単位、ページ遷移単位

Layout・Page・Async Component・DB・Server Action など複数のレイヤーが結合された状態で検証するシナリオ。コンポーネント単体で検証できる内容は対象外。

```
✅ 検索ページで条件を入力して一致する検索結果が表示される
✅ 登録フォームで入力して送信すると詳細ページに遷移する
❌ 登録フォームで必須入力を空にするとバリデーションエラーが表示される
   → コンポーネント単体で検証できるので small ブラウザテストで対応
```

## small ブラウザテスト

**対象：** クライアントコンポーネント、ユニバーサルコンポーネント、React Hook、ブラウザ用の関数

コンポーネント・Hook・関数単体で検証できるシナリオのうち、E2E テストでカバーしないもの。

```
✅ テキストフィールドが 0 文字・最大文字数超えのときエラーが表示される
✅ 送信中はボタンが無効化されローディング表示になる（アクションはモック）
✅ 送信失敗時にエラーメッセージが表示される（アクションはモック）
✅ ダイアログの開閉が正しく動作する
❌ 有効な入力で送信できる → 正常系フローは E2E テストでカバー
```

## medium サーバーテスト

**対象：**: Server Action, Route Handler、サーバー用の関数

E2E では代替しにくい、サーバーロジック固有のシナリオに使う。

```
✅ UI に表示されない DB 内部状態の検証（論理削除フラグ・タイムスタンプ等）
✅ 多数のパターンを網羅したいロジック
   （例：検索が姓・名・かな・電話番号それぞれでヒットすること）
   E2E は代表的なフローのみカバー、パターン網羅はサーバーテストで担う
✅ UI を持たないサーバーロジック（バックグラウンドジョブ・バッチ処理等）
❌ 登録フォームから保存できる → E2E で担保される
```

## small サーバーテスト

**対象：** Server Action、Route Handler、サーバー用の関数

DB・認証などの外部依存をモック化し、単体で検証できるシナリオ。

```
✅ 認証されていない場合に認証エラーが返る
✅ 一般ユーザーロールの場合に権限エラーが返る
✅ 必須フィールドが不正なフォーマットのときバリデーションエラーが返る
✅ ロジックがエラーを投げた場合にサーバーエラーが返る
❌ DB に値が正しく保存される → medium サーバーテストで対応
```
