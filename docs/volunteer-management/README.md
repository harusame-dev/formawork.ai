ボランティア来場管理システム — 要件整理
1. システム概要
ボランティアイベントにおける来場管理を行うWebシステム。
管理者（以下「ユーザー」）がイベントとボランティア情報を管理し、会場に設置した端末でボランティア自身が来場記録を行う。
技術スタック
項目技術フロントエンド / バックエンドNext.jsホスティングVercelデータベース / 認証Supabase

2. 利用者の定義
このシステムには2種類の利用者が存在する。混同を避けるため、用語を明確に分離する。
用語説明ユーザーシステムの管理者。管理画面を操作し、イベント作成・ボランティア登録・来場状況の確認を行うボランティアイベント参加者。会場の据え置き端末から来場記録を行う。管理画面にはアクセスしない

3. 設計上の重要な前提
3-1. ボランティアはイベント単位で管理し、イベント間で共有しない
同一人物であっても、イベントごとに個別の登録が必要。
背景： このシステムは「ボランティア会員をマスター管理し、参加イベントを紐づける」モデルではない。イベントごとに個別で参加を募り、そのイベント専用のボランティア名簿を管理するユースケースを想定している。
この判断のメリット：

イベントごとにデータが完結するためシンプル
イベント終了後のデータ管理（削除・アーカイブ）が容易
異なる主催者が別々のイベントを運用しても干渉しない

トレードオフ：

同一人物が複数イベントに参加する場合、都度登録が必要
ボランティア軸での横断検索（「この人は過去に何回参加したか」等）はできない


将来的にボランティアをマスター管理する方向に拡張する場合は、ボランティアテーブルをイベントから切り離し、中間テーブルで紐づける設計への変更が必要になる。

3-2. 来場記録は「押した回数分すべて記録」する
1日1回の制限は設けず、ボタンを押すたびにタイムスタンプを記録する。
背景： 午前の部・午後の部のように1日に複数回の出席確認が発生するケースに対応するため。システム側で回数を制限すると運用の柔軟性が失われる。
管理画面での表示： 来場状況一覧は「日付単位」で集約し、その日に1回以上記録があれば来場済みとする。詳細な打刻履歴はボランティア詳細画面で確認できる。
3-3. 来場ページはイベント固有URLで公開する
管理画面内のモード切替ではなく、独立したURLとして生成する。
背景： 据え置き端末での運用を想定しているため、管理画面のセッションに依存しない設計が望ましい。管理者がログアウトしても来場ページは独立して動作する必要がある。
セキュリティ対策： URLが外部に漏洩した場合に備え、URLの再生成（旧URL無効化）が可能。これにより、漏洩時は新しいURLを発行して旧URLを無効にするだけで対処できる。

4. 機能一覧
ユーザー（管理者）側
4-1. 認証

ログイン / ログアウト
Supabase Auth を利用

4-2. イベント管理
イベントの作成・編集・削除を行う。
項目説明イベント名任意の文字列イベント日複数日対応（例：1/11, 1/12, 1/13）イベント詳細自由記述
4-3. ボランティア管理
イベントに紐づくボランティアの登録・編集・削除を行う。
項目説明名前氏名ID数字6桁。イベント内で一意性別—イベント参加日イベント日の中から複数選択可
4-4. CSV取り込み
ボランティア情報をCSVファイルで一括登録する。
仕様内容単位イベント単位（1ファイル = 1イベント）バリデーションID形式（数字6桁）、必須項目チェック、ID重複チェックエラー時全件拒否。エラー内容を画面に表示
全件拒否の理由： 部分取り込みにすると「どこまで入ったか」の確認コストが運用者に発生する。名簿データは整合性が重要なため、修正して再アップロードしてもらう方が安全。
4-5. 来場状況一覧（イベント単位）
日付 × ボランティアのマトリクス形式で来場状況を表示する。
名前        1/11  1/12  1/13
───────────────────────────
根本将治     ◯    ✖️    ー
田中一郎     ◯    ◯    ◯
記号意味◯来場済み（参加予定日に来場記録あり）✖️未来場（参加予定日だが来場記録なし）ー非参加日（そもそも参加予定でない）
4-6. ボランティア詳細

登録情報（名前・ID・性別・参加日）の表示
来場記録の日時一覧（タイムスタンプ単位で全件表示）

4-7. 来場ページURL管理

イベント固有の来場ページURLを生成
URLの再生成が可能（再生成時に旧URLは無効化）


ボランティア側
4-8. 来場画面
イベント固有URLでアクセスする画面。認証は不要。

ID（数字6桁）を入力して本人照合
来場ボタンで来場を記録
押した回数分すべてタイムスタンプを記録

据え置き端末での運用を想定： 会場受付等にタブレットを設置し、ボランティアが順番にIDを入力して来場記録を行う。
