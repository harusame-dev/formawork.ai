### 基本方針

- Sandbox モードで実行することを想定しているため、明示的に allow の設定は行わない
- sandbox.excludedCommands に指定したコマンドは unsandboxed での実行を求めるプロンプトが表示されてしまうため、適宜 allow に設定する
- 破壊的なコマンドなど実行させたくないものは deny に登録する。
  - deny に登録しても迂回して実行されてしまう可能性があるので、実行されてしまっても問題ないようにネットワーク制限や環境の分離、権限管理などを行い、他段階の防衛を前提とする

## Sandbox

基本的に承認不要で自律的に開発させることを想定しおり、 破壊的な変更や重要な情報へのアクセス、ネットワーク制限を行うために
Sandbox 機能で環境の分離を行う
devContainer が代替の選択肢て有効性が高いが MAC だと IO が遅い問題があるため Sandbox で問題があるようなら再検討。

### 参考

https://code.claude.com/docs/en/claude-code-on-the-web

### 基本方針

- 有効にする
- network.allowedDomains には信頼ずみにドメインのみ指定
  - [Claude Code on the web の default allowed domains](https://code.claude.com/docs/en/claude-code-on-the-web#default-allowed-domains) に含まれているものはあらかじめ追加しておく
- excludedCommands は最低限にする

- filesystem.allowWrite は最低限にする

## HOOKS

### 参考

https://code.claude.com/docs/ja/features-overview#hooks

### 基本方針

#### コンテキストの注入

特定のタイミングや繰り返し注入したいコンテキストの注入

#### コマンドのブロック

Claude Code の設定で制御できない、もしくは厳格なブロックが必要な場合に使用する

#### lint の検証は行わない

Hook では Lint チェックを実施しない。

##### Post Edit（PostToolUse）で行わない理由

ファイル編集のたびにチェックが走るため、パフォーマンスへの影響が大きい。
チェック失敗時に exit code 2 で Claude に修正させる実装にした場合、コンテキストを消費し続ける。
最終的にチェックが通っていれば問題ないため、このタイミングでは実行しない。

##### Stop で行わない理由

参照：[フックリファレンス > StopおよびSubagentStop入力](https://code.claude.com/docs/ja/hooks#stopおよびsubagentstop入力)

> `stop_hook_active` は、Claude Code がすでに Stop フックの結果として続行している場合に `true` になる。無限ループを防ぐためにこの値をチェックすること。

````

① Claude が停止しようとする
→ Stop フック起動（stop_hook_active: false）
→ チェック失敗 → exit 2（ブロック）→ Claude が修正を試みる

② Claude が再度停止しようとする
→ Stop フック起動（stop_hook_active: true）
→ ループ防止のため即 exit 0（スルー）→ チェックせず通過

```

②の時点でチェックが走らないため、修正が正しかったかどうかを検証できない。
`stop_hook_active` は無限ループ防止のための設計であり、この制約を回避する手段は公式ドキュメントに存在しない。
```
````
