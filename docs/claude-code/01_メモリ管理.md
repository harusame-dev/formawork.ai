# メモリ管理

## ルート CLAUDE.md

<プロジェクトルート>/CLAUDE.md

### 参考

- https://github.com/abhishekray07/claude-md-templates/blob/main/principles.md
- https://www.humanlayer.dev/blog/writing-a-good-claude-md
- https://code.claude.com/docs/en/best-practices

### 基本方針

#### なるべく小さく保ち 100 行以内に保つ

CLAUDE.md が長くなりすぎると多くの命令を無視するようになる
100行以内に収まらない場合、モジュール CLAUDE.md / rules / agent_docs に分割を検討

特定のモジュール配下でのみ有効： モジュール CLAUDE.md
例：web モジュールでのみ必要 -> `apps/web/CLAUDE.md`

特定のファイルパスパターンでのみ有効： RULES
例：テストファイルでのみ必要 -> `.claude/rules/test.md`

特定のタスクや状況でのみ必要：agent-docs
例：ファイルを作成するときのみ必要 -> `agent-docs/命名規則.md`

#### ほとんどのタスクで毎回必要になる情報を記載する

特定のタスクやたまにしか必要にならない情報はコンテキストを無駄に消費するため記載しない

#### エージェントが調べればわかる内容は記載しない

エージェントは実装計画を立てるときに自律的に調査するため、 エージェントが調べればわかる内容を記載しない

例：使用しているライブラリは package.json を見ればわかる

#### 頻繁に変わる内容を記載しない

ドキュメントと実際の内容に乖離が生じたり、メンテナンスコストが発生するため

例：ライブラリの詳細なバージョン、特定のファイルのコードスニペット

#### `@file` は原則使用しない

ファイルを分割していてもセッション開始時に読み込まれるため、コンテキストが肥大する原因になる

#### 記載しなくても上手くいくことを書かない

最初から念の為書いておくのではなく、書かないと上手くいかなかった時に記載
モデルのアップデートなどで書かなくても上手くいくようになったら削除する

#### linter / formatter で担保できるコードスタイルは書かない

AIに任せるのは高コストで linter / formatter の auto fix で短時間、正確に修正できる

### 記載内容

#### リポジトリ概要

リポジトリの簡単な説明

#### フォルダ構成

大まかな構成のみ（詳細はエージェントが調べる）  
モノレポの場合はパッケージ名も記載

#### コマンド

エージェントが開発で使うプロジェクト固有のコマンドのみ

#### プロジェクト固有の情報

- 開発サーバー情報
- ブランチ命名規則

#### AI エージェントのワークフロー

- タスク実行時にやること
- 完了基準

#### 参照ドキュメント

- `agent_docs` 配下のフォルダを参照すること、必要に応じて読み込むことのみ記載
- 実際のファイル一覧は hooks によりセッション開始時に注入（ファイル増減のたびの手動更新を防ぐため）

## モジュール CLAUDE.md

apps/web/CLAUDE.md など

### 基本方針

#### 特定のモジュール配下でのみ有効な場合に使用する

RULES でも同じことができるが判断コストを避けるためモジュール CLAUDE.md で統一する

#### 小さいプロジェクトでルート CLAUDE.md が十分に小さい場合は分けなくても良い

コンテキストを小さく保つための分割なので、ルート CLAUDE.md が十分に小さく保てるのであれば分けなくても OK

## RULES

.claude/rules/testing.md など

### 参考

- https://code.claude.com/docs/ja/memory#claude/rules/-でルールを整理する

### 基本方針

#### ファイルパスだけでそのファイルが何なのか表せるようにする

ファイルパスによって渡すコンテキストを制御しやすいように拡張子のプレフィクスやフォルダ構成によってそのファイルが何なのか判別しやすいようにする

例： サーバーアクション -> create-user.action.ts

### ファイルパス規約

フレームワークで規定されていない独自ルール

| ファイル種別               | 適用パターン                             |
| -------------------------- | ---------------------------------------- |
| サーバーアクション         | `*.action.ts`                            |
| サーバーコンポーネント     | `*.server.tsx`                           |
| クライアントコンポーネント | `*.client.tsx`                           |
| ユニバーサルコンポーネント | `*.universal.tsx`                        |
| react hook                 | `*.hook.ts`                              |
| ブラウザテスト             | `*.{small,medium}.browser.test.{ts,tsx}` |
| サーバーテスト             | `*.{small,medium}.server.test.ts`        |
| e2eテスト                  | `*.{small,medium}.e2e.test.ts`           |

## agent-docs

### 基本方針

- CLAUDE.md には `agent-docs` を必要に応じて読み込むように記載し、エージェントが自律的に判断して読み込む

Agent skill でもリファレンス型のスキルを作成して同じようなことができるが、自律的な読み込み精度に課題があり、必要な時に呼び出されない可能性が現時点では高い。
そのため CLAUDE.md にドキュメントのインデックスを含めて自律的に読み込むようにする。
https://vercel.com/blog/agents-md-outperforms-skills-in-our-agent-evals を参照

なお、ドキュメントのインデックスは手動でメンテナンスすることを避けるため、 hooks でセッション開始に自動的に注入するようにする

#### 特定の状況、タスクで必要になるリファレンスを記載する

命名規則やアーキテクチャのように特定のファイルパスや毎回必要になるわけでない情報などは agent-docs 配下に markdown でまとめて記載する。

#### ファイル名は日本語で、名前だけで内容がわかるようにする

hooks で注入するため、そのファイルが何なのかを説明しなくていいようにわかりやすい簡潔な名前をつける

#### 適切に分割し、関連する情報のみをコンパクトに記載する

不必要なコンテキストを読み込まないように必要な情報だけをコンパクトに記載し、分割して記載する
