# チケット 02: 認証機能（ログイン／ログアウト）

## 概要

既存のログイン・ログアウト機能を基本的にそのまま流用しつつ、
LP 削除に伴うルーティング変更と、パスワード変更画面の調整を行う。

## 背景

ログイン・ログアウト・セッション管理は Supabase Auth で実装済みであり、
認証の仕組み自体は変更不要。ただし LP を削除するため、
未認証時のルーティングを見直す必要がある。

## 現状

| 機能 | 実装状況 | 変更要否 |
|---|---|---|
| ログイン画面 (`/login`) | 実装済み | 変更なし |
| ログイン API（Supabase Auth） | 実装済み | 変更なし |
| セッション管理（SSR Cookie） | 実装済み | 変更なし |
| ログアウト処理 | 実装済み | 変更なし |
| 未認証時リダイレクト | 実装済み（middleware） | 要確認 |
| パスワード変更（本人操作） | 実装済み（`/settings/password`） | 要確認 |
| ランディングページへのリダイレクト | 実装済み | **変更必要** |

## 変更内容

### 1. LP 削除後のルーティング修正

- `apps/web/app/(public)/` 配下のルートを整理する
- チケット 00 で LP を削除した後、`/` ルートが存在しなくなるため、
  未認証ユーザーが `/` にアクセスした場合のリダイレクト先を確認・修正する
- 認証済みユーザーのデフォルト遷移先は `/` (ダッシュボード) のまま維持する

### 2. middleware の確認

- `apps/web/middleware.ts` のパスマッチャーに LP のパスが含まれていないか確認する
- 不要なパスの除外設定があれば修正する

### 3. パスワード変更画面

- `/settings/password` は **そのまま流用**（本人によるパスワード変更）
- 管理者による他ユーザーのパスワード再設定は **チケット 03** で実装する

## 変更しないもの

- `apps/web/features/auth/login/` — 変更なし
- `apps/web/features/auth/logout/` — 変更なし
- `apps/web/features/auth/change-password/` — 変更なし
- `apps/web/features/auth/user/role.ts` — 変更なし
- `apps/web/features/auth/get-user-role.ts` — 変更なし
- `apps/web/features/auth/get-user-staff-id.ts` — 変更なし（staffId をそのまま userId として扱う）
- Supabase Auth の設定（app_metadata の role / staffId）— 変更なし

## 完了条件

- [ ] 未認証状態で任意のページにアクセスすると `/login` にリダイレクトされる
- [ ] ログイン後にダッシュボード（`/`）に遷移する
- [ ] ログアウト後に `/login` に遷移する
- [ ] LP 削除後もビルドエラー・ルーティングエラーが発生しない
